# LLM-Native Per-Turn Memory Protocol

**Date:** 2026-06-07  
**Status:** Approved for implementation  
**Replaces:** Stop hook index heuristic (regex tier matching in `lib/index-heuristic.sh` / `indexCandidates()`)

---

## 1. Problem

The Stop hook `index-heuristic.sh` (and its TypeScript equivalent `indexCandidates()`) detects memory-worthy signals using regex tier matching against the raw user message and last assistant response. This produces two failure modes:

1. **Poor subject quality** — subjects are generated by slugifying the raw user message (`echo "$user" | tr ... | cut -c1-40`), producing labels like `event-2026-05-30-dubai-hotel-cancell` that are semantically meaningless and pollute the subject registry.

2. **Structural under-capture** — the heuristic operates on two text blobs at session end with no conversational context. Signals that span multiple turns, emerge implicitly, or require judgment (e.g. a correction embedded in a follow-up question) are missed.

The per-turn SKILL.md protocol (Plane 2) already runs LLM inference with full context. The heuristic was the safety net for what the protocol missed. This design eliminates the heuristic and makes the per-turn protocol the complete and authoritative write path.

---

## 2. Goals

| ID | Goal |
|----|------|
| G1 | All memory writes use full conversational context — no regex, no slug subjects |
| G2 | Coverage of all five signal categories guaranteed by a two-phase protocol |
| G3 | Stop hook becomes completion-only — no new writes at session end |
| G4 | Identical protocol across Claude Code, Cursor, and OpenClaw plugins |
| G5 | Noise tolerance maintained via confidence discipline, not write suppression |

---

## 3. Signal Categories

The protocol guarantees coverage of exactly these five categories per turn:

| # | Category | Write target | Example |
|---|----------|--------------|---------|
| 1 | **Preference** | `memory_write` | "I always want tests before code" |
| 2 | **Correction** | `memory_write` | User pushed back on a proposed approach |
| 3 | **Project constraint** | `memory_write` | Deadline, policy, scope limit, freeze |
| 4 | **Architectural decision** | `memory_write` | Design choice, technology, pattern selected |
| 5 | **Deferred intention** | `set_intention` | "We'll tackle X after Y is merged" |

---

## 4. Protocol Design

### 4.1 Trigger

After every agent response, regardless of whether the agent believes something durable happened. The check is cheap; missed captures are not recoverable.

### 4.2 Phase 1 — Open Reflection

The agent considers the full conversation up to the current turn (all n turns of context) and asks:

> *What emerged in this turn that future-me wouldn't know from reading the code or git history?*

This is intentionally open-ended. The agent uses holistic judgment — not pattern matching — to surface candidates. Phase 1 may produce zero, one, or multiple candidates.

### 4.3 Phase 2 — Category Audit (backstop)

For each of the five categories, if Phase 1 did not already produce a candidate for that category, the agent checks explicitly:

1. **Preference** — Did the user state or confirm how they like something done?
2. **Correction** — Did the user push back? Did the agent get something wrong?
3. **Project constraint** — Did a deadline, requirement, or policy emerge?
4. **Architectural decision** — Was a design or technology choice made or confirmed?
5. **Deferred intention** — Was something identified as "do later" or "follow up on"?

Phase 2 is the under-writing fix. It does not override Phase 1 — it catches what Phase 1 missed.

### 4.4 Write Routing

```
Categories 1–4  →  memory_write
Category 5      →  set_intention
Nothing found   →  skip (no write)
```

### 4.5 Confidence Calibration

| Evidence | Confidence |
|----------|-----------|
| User stated explicitly ("I prefer…", "always…") | 0.90–0.95 |
| User confirmed when agent asked | 0.80–0.85 |
| Inferred from behavior or implicit context | 0.65–0.75 |
| Uncertain / speculative | Skip |

The agent **must not** write at 0.90+ for inferred signals. agent-brain's recall ranking weights confidence directly — inflated confidence on noise causes it to surface ahead of legitimate writes and blocks supersession.

### 4.6 Subject Naming

Subjects must be short canonical labels derived from conversation context:

- **Valid:** `testing-approach`, `auth-middleware`, `code-style`, `deploy-policy`
- **Invalid:** `event-2026-05-30-dubai-hotel-cancell`, `2026-06-07-user-asked-about-tabs`, any slug of raw message text

When a subject already exists in the agent's recall context, reuse it. Create a new subject only when the topic is genuinely not covered by an existing one.

## 5. Session Lifecycle (After)

```
Session Start
  → memory_search (parallel)
  → check_intentions (parallel)
  → retrieve_skills_for_context (parallel)
  → inject recall + intentions + skill block into first prompt

Per-turn (each user message)
  → recall hook: memory_search + check_intentions
  → save triggered intention IDs to session state file

Per-turn (after each agent response)  ← PRIMARY WRITE PATH
  → Phase 1: open reflection (full context)
  → Phase 2: category audit (5-category backstop)
  → memory_write (categories 1–4) or set_intention (category 5)

Stop / agent_end
  → Phase 1: complete triggered intention IDs → delete session state file
  → Phase 2: (removed — no heuristic, no new writes)
```

## 7. Noise Tolerance

Removing the heuristic does not increase net noise because:

1. **Dedup at write time** — identical content hash is rejected by `write/writer.go`
2. **Supersession** — higher-authority write on same subject cascades staleness on weaker priors
3. **Recall ranking** — `engine.go` weights confidence in RRF scoring; 0.65–0.75 inferred memories naturally sink below 0.90+ stated facts

The protocol relies on confidence discipline (§4.5) to keep inferred noise in the low-confidence band where these mechanisms clean it up automatically. Systematic high-confidence misclassification remains the one failure mode that bypasses all three mechanisms — the protocol addresses this with explicit calibration anchors.

## 8. Testing

| Scenario | Expected outcome |
|----------|-----------------|
| User states preference explicitly | Phase 1 produces candidate; written at 0.90+ |
| Correction spans two turns | Phase 1 uses full context; captured in turn where correction completes |
| Nothing durable in turn | Phase 2 returns nothing for all 5 categories; no write |
| Deferred item mentioned | Category 5 check fires; `set_intention` called |
| Subject matches existing registry entry | Agent reuses existing subject label |
| Stop hook fires | Only `complete_intention` calls; no `memory_write` |

---

## 9. Out of Scope

- Changing the session-start recall flow
- Changing the intention lifecycle (check → triggered IDs → complete)
- Adding new signal categories beyond the five defined in §3
- Automated cleanup of previously written heuristic-produced subjects (a separate operation)
